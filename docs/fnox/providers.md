# Провайдеры

## Что такое провайдер

Провайдер — это конфигурация шифрования. Он определяет **каким ключом** шифруются секреты и **кто** сможет их расшифровать.

В `fnox.toml` провайдер выглядит так:

```toml
[providers.main]
type = "age"
recipients = [
    "age1abc123...",  # Иван
    "age1def456...",  # Мария
]
```

`recipients` — список публичных ключей. Секрет, зашифрованный этим провайдером, сможет прочитать только тот, чей публичный ключ есть в этом списке **и** у кого есть соответствующий приватный ключ.

## Несколько провайдеров

Можно создать несколько провайдеров с разными списками доступа. Это удобно когда в команде есть разные роли:

```toml
[providers.frontend]
type = "age"
recipients = [
    "age1...",  # Саша (фронтенд)
    "age1...",  # Дима (фронтенд)
]

[providers.backend]
type = "age"
recipients = [
    "age1...",  # Дима (бэкенд)
    "age1...",  # Марина (бэкенд)
    "age1...",  # CI/CD
]

[secrets]
VITE_API_URL  = { provider = "frontend", value = "..." }  # видит фронт
DATABASE_URL  = { provider = "backend",  value = "..." }  # видит бэк
JWT_SECRET    = { provider = "backend",  value = "..." }  # видит бэк
```

Саша клонирует репо — видит только `VITE_API_URL`. `DATABASE_URL` для неё останется недоступным, даже если она попробует его расшифровать — математика не позволит.

## Почему нельзя просто добавить себя в чужой провайдер

Добавить свой публичный ключ в `recipients` другого провайдера технически возможно. Но это ничего не даст.

Зашифрованные значения секретов в `fnox.toml` создавались **под конкретный список recipients**. Чтобы они стали доступны новому участнику — нужно перешифровать их заново с учётом его ключа. Это делает команда `fnox reencrypt`, и для её запуска нужен приватный ключ **уже существующего** участника.

То есть: добавить себя можно, но расшифровать — нет. Реальный доступ выдаёт только тот, у кого он уже есть.
